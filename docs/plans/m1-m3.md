## M1: フィードバック学習ループ

### 1-1. `scripts/ingest.mjs` — `candidates` サブコマンド拡張
- **有効判定の定義**（REQUIREMENTS §5）: item_idごとに `decided_at` 最大（同値なら `id` 最大）のイベントのverdict。`undo` は「未判定」扱い。
- feedback_events × items を集計し、学習プロファイルを導出して `data/learned-profile.json` に書き出し、candidates出力の `learned_profile` フィールドにも同内容を含める。**有効判定が1件もない場合は learned-profile.json を書かず `learned_profile: null`**（従来挙動）。
- learned-profile.json スキーマ:
```json
{
  "generated_at": "ISO8601",
  "based_on": {"favorite": 5, "not_interested": 3, "skip": 2},
  "liked_topics":    [{"topic": "mcp", "count": 3}],
  "disliked_topics": [{"topic": "release-watch", "count": 2}],
  "liked_sources":   [{"source": "hn", "count": 4}],
  "disliked_sources":[{"source": "agents", "count": 2}],
  "serendipity_promoted": ["kani", "formal-verification"]
}
```
  - topicsは items.topics_json から。count降順、各最大15件。両リストに同一topicが出る場合は差し引かず両方に出す（解釈はClaude側の仕事）
  - `serendipity_promoted`: `is_serendipity=1` かつ有効判定が `favorite` のitemのtopics（重複排除）

### 1-2. `scripts/sync-feedback.mjs`（新規）
- クラウド（M3）のフィードバックをローカルDBに取り込む。`data/cloud.json`（`{"url": "https://winnow.xxx.workers.dev", "key": "..."}`）を読む。**存在しなければ "cloud not configured, skip" をstderrに出して exit 0**
- `GET {url}/api/feedback/export?since=<ISO>` を `X-Winnow-Key: <key>` 付きで呼ぶ（Node標準のfetch）。sinceはローカル feedback_events の `MAX(decided_at)`（なければ '1970-01-01T00:00:00Z'）
- 返ってきたeventsのうち、`(item_id, decided_at, verdict)` が完全一致する行がローカルに無いものだけINSERT。`{imported: N}` をstdoutにJSON出力

## M2: 定期実行 + アーカイブ索引

### 2-1. `scripts/daily.sh`（新規）
- `#!/usr/bin/env bash`、`set -uo pipefail`（-eは使わず自前ハンドリング）
- `export PATH="$HOME/.local/bin:/opt/homebrew/bin:/usr/bin:/bin"`（claude=~/.local/bin, node=/opt/homebrew/bin, jq=/usr/bin）
- リポジトリルートに cd し、`mkdir -p data/logs`、ログを `data/logs/daily-YYYY-MM-DD.log` に追記
- 実行: `claude -p "/winnow" --permission-mode acceptEdits` （stdout/stderrをログへ）
- 成否で macOS 通知: `osascript -e 'display notification "..." with title "Winnow"'`（成功時はレポートURL http://127.0.0.1:8765/YYYY-MM-DD/report.html を本文に）
- 同日ログが既に「SUCCESS」を含む場合は二重実行しない（スリープ復帰で多重発火する場合の防御）

### 2-2. `launchd/` （新規ディレクトリ、plist 2本+インストーラ）
- `launchd/com.winnow.daily.plist`: `/bin/bash <ABS>/scripts/daily.sh` を `StartCalendarInterval {Hour:7, Minute:0}` で実行。StandardOut/ErrorPath は `<ABS>/data/logs/launchd-daily.log`。`<ABS>` はリポジトリの実パス（後にテンプレート化し、install-launchd.shが実行時に置換する方式へ変更）
- `launchd/com.winnow.serve.plist`: `/opt/homebrew/bin/node <ABS>/scripts/serve.mjs --no-timeout` を `KeepAlive true` / `RunAtLoad true` で常駐
- `scripts/install-launchd.sh`: 2つのplistを `~/Library/LaunchAgents/` にコピーし `launchctl bootout gui/$UID/<label>`(失敗無視)→`launchctl bootstrap gui/$UID <plist>`。`--uninstall` オプションで解除

### 2-3. `scripts/index.mjs`（新規）
- `output/*/stories.json` を走査し `output/index.html` を生成（新しい日付順）。各行: 日付 / ストーリー数(🎲数) / macro_summary 1行目 / report.htmlへのリンク
- デザインはreport.htmlと同じトークン（page/ink/gold、mono eyebrow）。自己完結1ファイル・ダーク対応。render.mjsのesc関数相当を再実装してXSS対策

### 2-4. `.claude/settings.local.json`（winnow直下、新規）
headless実行（`claude -p`）が権限プロンプトで止まらないための事前許可:
```json

  }

```

## M3: Cloudflare配信（生成はローカルのまま）

### 3-1. `cloud/` — Hono Worker（TypeScript）
構成: `cloud/package.json`（hono。devDeps: wrangler, @cloudflare/workers-types）、`cloud/wrangler.jsonc`、`cloud/schema.sql`、`cloud/src/index.ts`。

`wrangler.jsonc`:
```jsonc
{
  "name": "winnow",
  "main": "src/index.ts",
  "compatibility_date": "2026-07-01",
  "kv_namespaces": [{ "binding": "REPORTS", "id": "__KV_ID__" }],
  "d1_databases": [{ "binding": "DB", "database_name": "winnow", "database_id": "__D1_ID__" }],
  "observability": { "enabled": true }
}
```
（`__KV_ID__`/`__D1_ID__` は依頼側が置換）

`schema.sql`（D1用。itemsはローカルに残すのでFKなし）:
```sql
CREATE TABLE IF NOT EXISTS feedback_events (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  item_id TEXT NOT NULL,
  cluster_id TEXT,
  verdict TEXT NOT NULL CHECK (verdict IN ('favorite','not_interested','skip','undo')),
  decided_at TEXT NOT NULL,
  run_id INTEGER
);
CREATE INDEX IF NOT EXISTS idx_feedback_item ON feedback_events(item_id, decided_at);
```

`src/index.ts`（Hono）— ルート仕様:
- **認証ミドルウェア**（`/api/health` 以外の全ルート）: 次のいずれかで認証。①`X-Winnow-Key` ヘッダ == env.WINNOW_KEY、②cookie `wk` == WINNOW_KEY、③クエリ `?key=` == WINNOW_KEY（このとき `Set-Cookie: wk=<key>; HttpOnly; Secure; SameSite=Lax; Max-Age=31536000; Path=/` を付けて以後クエリ不要に）。不一致は401でJSON `{error:"unauthorized"}`
- `GET /api/health` → `{ok:true, version:"0.2.0"}`（認証不要）
- `GET /` → KVを `prefix: "report:"` でlist、日付降順のアーカイブ索引HTML（インラインCSS、report.htmlと同系トークン、ダーク対応）
- `GET /r/:date` → KV `report:<date>` を `text/html; charset=utf-8` で返す。無ければ404
- `PUT /api/reports/:date` → リクエストbody(text)をKV `report:<date>` にput。`{ok:true}`。dateは `YYYY-MM-DD` 形式のみ許可
- `POST /api/feedback` → serve.mjsと同一コントラクト: `{run_id, cluster_id, item_ids[], verdict(4値)}`、`decided_at` はサーバー付与、item_idごとに1行INSERT → `{ok:true, recorded:N}`。不正は422
- `GET /api/feedback/summary` → verdict別・run別集計
- `GET /api/feedback/export?since=<ISO>` → `decided_at > since` の全行を `{events:[{item_id, cluster_id, verdict, decided_at, run_id}]}` で返す（decided_at昇順、最大5000行）

### 3-2. `scripts/render.mjs` 変更（2点）
- 環境変数 `WINNOW_API_BASE` があれば埋め込む `apiBase` をその値にする（空文字列も有効値。空文字列=同一オリジン相対）。未設定時は従来どおり `http://127.0.0.1:<port>`
- 第2引数 `--out <dir>` があれば report.md/report.html の出力先をそのディレクトリにする（デフォルトは従来どおりstories.jsonの隣）

### 3-3. `scripts/publish.mjs`（新規、依存ゼロ）
- 引数: `<stories.json>`。`data/cloud.json` が無ければ "cloud not configured, skip" で exit 0
- 手順: ①一時ディレクトリに `WINNOW_API_BASE="" node scripts/render.mjs <stories> --out <tmp>` でクラウド用HTMLを生成（child_process.execFileSync）、②`PUT {url}/api/reports/<date>`（`X-Winnow-Key`、body=HTML）、③成功で `{ok:true, url:"{url}/r/<date>"}` をstdoutへ
- dateは stories.json の `date` フィールドを使う

### 3-4. テンプレート `templates/report.html` 微修正
- `fetch(\`${data.apiBase}/api/feedback\`)` が `apiBase=""`（相対）でも動くことを確認（現状の文字列連結で動くはずなので、動くならコード変更不要）

### 3. `scripts/fetch/*.sh`（9本）

共通仕様:
- 引数なしで実行可能。`config/sources.json` を jq で読む（スクリプトからの相対パス解決: `$(dirname "$0")/../../config/sources.json`）
- **stdoutに共通スキーマのJSON配列**を出力: `[{source, url, title, author, published_at(ISO8601), engagement:{...ソース固有キー}, raw_tags:[]}]`
- 該当0件でも空配列 `[]` を出力して exit 0。エラー時はstderrにメッセージ+非ゼロexit
- `set -euo pipefail`。取得は `curl -sf --max-time 30`。件数上限 `max_items_per_source`
- engagementキーはソース別: hn=`{points, comments}` / hatebu=`{users}` / qiita=`{stocks, likes}` / lobsters=`{points, comments}` / devto=`{reactions, comments}` / ghtrend・agents・zenn・reddit=取れる範囲で（無ければ `{}`）

| スクリプト | 内容 |
|---|---|
| `hn.sh` | Algolia 2系統: `search?tags=front_page&hitsPerPage=50` と、focus_keywordsごとに `search_by_date?query=<urlencode(kw)>&tags=story&numericFilters=points>10,created_at_i><now-48h epoch>`。objectIDを `engagement.hn_object_id` に保持（コメント取得で使う） |
| `zenn.sh` | `zenn.dev/feed` + zenn_topicsごとの `zenn.dev/topics/{t}/feed`。RSS(XML)のパースはsed/awkで最低限（title/link/pubDate/creator） |
| `qiita.sh` | `qiita.com/api/v2/items?query=<qiita_query>&per_page=50`。`QIITA_TOKEN` があれば `Authorization: Bearer` を付与。fetch_window_hoursでフィルタ |
| `hatebu.sh` | `b.hatena.ne.jp/hotentry/it.rss`（RDF/RSS1.0）。ブクマ数は `hatena:bookmarkcount` 要素 |
| `ghtrend.sh` | `mshibanami.github.io/GitHubTrendingRSS/daily/all.xml` |
| `reddit.sh` | subredditsごとに `old.reddit.com/r/{sub}/.rss`（Atom）。UA明示、リクエスト間 `sleep 2`。**403/429は空配列+exit 0（stderrに理由）** — ベストエフォートソースのため |
| `lobsters.sh` | `lobste.rs/hottest.json` |
| `agents.sh` | github_release_reposごとに `api.github.com/repos/{r}/releases?per_page=5`（UA必須）+ github_commit_feedsごとに `github.com/{r}/commits.atom` |
| `hn_comments.sh <objectID>` | `hn.algolia.com/api/v1/items/<id>` から上位コメント最大20件を `[{author, text(タグ除去), points}]` で出力 |

### 4. `scripts/ingest.mjs`（サブコマンド: ingest / candidates / finalize）

`node:sqlite` 使用。DBパス `data/winnow.db`（`WINNOW_DB` 環境変数で上書き可）。初回起動時にREQUIREMENTS.md §5のDDLでスキーマ作成。

- **`ingest <file...>`**: raw JSONを読み、URL正規化（§4.2の5規則を厳密に実装: scheme/host小文字化→フラグメント除去→`utm_*`,`gclid`,`fbclid`,`ref_src`,`source`除去→クエリキー昇順ソート→末尾スラッシュ除去）→ sha256でID化 → items upsert（既存IDに別ソースが来たら `engagement_json` のソース別マップに追記し `source_count` 更新）→ 新規runを作成し `first_seen_run` 設定 → **stdoutに `{"run_id": N, "ingested": N, "new": N}` をJSON出力**
- **`candidates --run <id>`**: `last_shown_run IS NULL` のitemを対象に、**quality_score = 同一source内でのengagement主要値の百分位順位(0-100)** を算出して付与（sourceごとの主要値: hn=points, hatebu=users, qiita=stocks, lobsters=points, devto=reactions, それ以外=順位なし→50）。stdoutにJSON出力: `{run_id, candidates: [{id, url, title, source, published_at, engagement, source_count, quality_score, raw_tags}], learned_profile: <data/learned-profile.jsonがあれば内容、なければnull>}`
- **`finalize <stories.json> --run <id>`**: 各story×itemsについて `translated_title, summary, cluster_id, match_score, topics_json, is_serendipity` を書き戻し、`last_shown_run = run_id` を設定。掲載itemがDBに無い場合はエラーで落とさずstderrに警告

### 5. `scripts/validate.mjs <stories.json>`

REQUIREMENTS.md §7のvalidate基準表を実装。全チェックを走らせ、違反を `[{rule, cluster_id?, message}]` でstdoutにJSON出力。違反0なら exit 0、あれば exit 1。

### 6. `scripts/render.mjs <stories.json>`

- `report.md`: マクロ要約 → ストーリー（見出し/メタ行/要約/選定理由/セクション/リンク） → 取得状況、のMarkdown
- `report.html`: `templates/report.html` の `/*__WINNOW_DATA__*/` プレースホルダに stories.json + `{apiBase: "http://127.0.0.1:" + (WINNOW_PORT||8765)}` をJSONとして埋め込み、同ディレクトリに出力

### 7. `scripts/serve.mjs`

REQUIREMENTS.md §4.7のAPIコントラクトを厳密に実装:
- `node:http`、bind **127.0.0.1** のみ、ポート `WINNOW_PORT`（既定8765）
- 起動時: 既存 `GET /api/health` が応答すれば「reuse existing server」とstderrに出して exit 0。EADDRINUSEかつhealth不通なら非ゼロexit
- `GET /api/health` → `{ok, version: "0.1.0", db}` / `POST /api/feedback`（body: `{run_id, cluster_id, item_ids[], verdict}`、verdictは4値のみ許可、`decided_at` はサーバー付与、item_idごとに1イベントINSERT）→ `{ok, recorded}` / `GET /api/feedback/summary` → verdict別・run別集計 / それ以外 → `output/` 配下の静的配信（パストラバーサル対策必須）
- `/api/*` に `Access-Control-Allow-Origin: *` + OPTIONSプリフライト（204）
- アイドルタイムアウト120分で自動終了（`--no-timeout` で無効化）

### 8. `templates/report.html`

自己完結型1ファイル（CSS/JSインライン、外部リクエストは埋め込まれたapiBaseへのfetchのみ）。`prefers-color-scheme` でライト/ダーク両対応。日本語UI。

- **ヘッダ**: 日付、マクロ要約3行、ソース内訳の帯グラフ（stories[].items[].sourceの構成比を色分け1本バーで）、モード切替タブ（📖/🃏）
- **📖 ドキュメントモード**: ストーリーごとにカード。見出し/メタバッジ行（ソースアイコン絵文字・source_count・engagement・公開日・composite_score、is_serendipityなら🎲）/要約/選定理由/可変セクション（`<details>` で開閉、存在するものだけ）/関連リンク。フィードバック済みitemはバッジ表示
- **🃏 トリアージモード**: カードスタック。上のカードをドラッグ（Pointer Events）で右=⭐favorite/左=✋not_interested、閾値未満は戻る。キーボード `→`/`←`/`↓`(skip)/`z`(undo)。ボタンUIも併設（⭐/✋/スキップ/戻す）
  - 判定→ `POST {apiBase}/api/feedback`（story内の全item_ids + cluster_id + run_id + verdict）
  - undo → 直前判定と同じitem_idsで `verdict:'undo'` をPOSTし、カードを山に戻す
  - **送信失敗時はlocalStorageのキュー（キー: `winnow-feedback-queue`）に積み、ページロード時とオンライン復帰時に再送**
  - 進捗表示（残りN枚）、全カード終了時はサマリ（⭐n / ✋n / skip n）
- グラフ・アイコンはCSS/絵文字/インラインSVGで実現（画像ファイル・外部フォント禁止）

### 9. `fixtures/` + `scripts/smoke.sh`

- `fixtures/raw-{hn,zenn}.json`（共通スキーマのサンプル各3件、実在URLを模したダミー）と `fixtures/stories.sample.json`（§5.1スキーマ準拠、2ストーリー、うち1つはsectionsフル）
- `smoke.sh`: `WINNOW_DB=/tmp/winnow-smoke.db` で ingest→candidates→(fixtures/stories.sample.jsonのrun_id/item idを実IDに差し替え)→validate→finalize→render を通し、serve.mjsをバックグラウンド起動して health/feedback POST/summary をcurlで検証、`sqlite3` かNodeワンライナーで feedback_events の行数を確認して全結果をPASS/FAILで出力。終了時にサーバーを止める
- `.gitignore`: `data/`, `output/`, `/tmp` 系
